= GRASP (General Responsibility Assignment Software Patterns)
:toc:
:toclevels: 2

*GRASP* — *G*eneral *R*esponsibility *A*ssignment *S*oftware *P*atterns — was introduced by *Craig Larman* in _Applying UML and Patterns_ (1997).

Where GoF design patterns answer "what shape should this collaboration have?", GRASP answers a more foundational question:

[quote]
Which class should be responsible for this behaviour?

It is a set of nine heuristics — *not laws* — for *assigning responsibilities* to classes. Together they form the conceptual bedrock that SOLID later refined.

== What is a responsibility?

In software, a *responsibility* is an obligation to perform a task or to know information. It has two forms:

* *Behaviour* — doing something.
* *Data* — knowing something.

GRASP gives you a vocabulary for deciding *which class* should hold each of those.

== The nine patterns

Each pattern is stated as a *Problem* and a *Solution* — the form Larman uses in the book.

=== 1. Information Expert

[cols="1,4", options="header"]
|===
| | Statement

| *Problem*
| What is a basic principle by which to assign responsibilities to objects?

| *Solution*
| Assign a responsibility to the class that has the information needed to fulfil it.
|===

If `Order` knows its line items and their prices, then `Order.getTotal()` belongs on `Order` — not on `OrderTotalCalculator`. Putting behaviour next to its data minimises data shuttling and keeps cohesion high.

*The single most-applied GRASP pattern.* Most "where does this method go?" questions are answered by Information Expert.

=== 2. Creator

[cols="1,4", options="header"]
|===
| | Statement

| *Problem*
| Who creates object A?

| *Solution*
| Assign class B the responsibility to create object A if one (or more) of these is true:

* B contains or compositely aggregates A
* B records A
* B closely uses A
* B has the initialising data for A
|===

So `Order` creates `OrderLineItem` (it aggregates them). A `ShoppingCart` creates `CartItem`. Avoid sprinkling factories everywhere when the natural container can own creation. If `A` cannot live without `B`, it's a strong signal that `B` should own creation of `A`.

=== 3. Controller

[cols="1,4", options="header"]
|===
| | Statement

| *Problem*
| What first object beyond the UI layer receives and coordinates ("controls") a system operation?

| *Solution*
| Assign the responsibility to an object representing one of:

* The overall *system*, *root object*, *device*, or a *major subsystem* (variations of a _facade controller_).
* A *use case scenario* within which the system operation occurs (a _use case / session controller_).
|===

The Controller is the *first object beyond the UI layer* that receives and coordinates a system operation. *It does not do the work itself — it delegates.* This is why "fat controllers" are a smell: the Controller should be thin and delegate to domain Experts.

This pattern is the direct ancestor of *MVC's controller* and of *Clean Architecture's use case interactors*.

=== 4. Low Coupling

[cols="1,4", options="header"]
|===
| | Statement

| *Problem*
| How to reduce the impact of change? How to support low dependency and increased reuse?

| *Solution*
| Assign responsibilities so that (unnecessary) coupling remains low. Use this principle to *evaluate alternatives*.
|===

Same idea as xref:ROOT:software-architecture/principles/coupling-and-cohesion.adoc[coupling-and-cohesion] — minimise the dependencies a class has on other classes. When you have multiple valid options for where a responsibility could go, prefer the placement that creates fewer connections.

=== 5. High Cohesion

[cols="1,4", options="header"]
|===
| | Statement

| *Problem*
| How to keep objects focused, understandable, manageable, and as a side effect support Low Coupling?

| *Solution*
| Assign a responsibility so that cohesion remains high. Use this to *evaluate alternatives*.
|===

Each class should have a focused, related set of responsibilities. *This is the direct GRASP-era precursor to SRP.* Larman wrote about cohesion years before Martin formalised "one reason to change".

Low Coupling and High Cohesion are the *evaluative* patterns — you apply the others, then check your design against these two.

=== 6. Polymorphism

[cols="1,4", options="header"]
|===
| | Statement

| *Problem*
| How to handle alternatives based on type?

| *Solution*
| When related alternatives or behaviours vary by type (class), assign responsibility for the behaviour — using *polymorphic operations* — to the types for which the behaviour varies.
|===

[source,typescript]
----
// Smell — type checking
if (animal.type === "dog") { dog.bark(); }
else if (animal.type === "cat") { cat.meow(); }

// GRASP applied — polymorphic operation
animal.speak();
----

Variants in behaviour → variants in subtype, not in `if`-chains. This is the GRASP-era precursor to *OCP* and the foundation for *LSP*.

=== 7. Pure Fabrication

[cols="1,4", options="header"]
|===
| | Statement

| *Problem*
| What object should have the responsibility, when you do not want to violate High Cohesion and Low Coupling, but solutions offered by other principles are not appropriate?

| *Solution*
| Assign a highly cohesive set of responsibilities to an artificial or convenience class that does not represent a problem-domain concept.
|===

Sometimes Information Expert wants to put behaviour on a domain class (`Order.save()`) — but persistence has nothing to do with the domain meaning of "order". So you *fabricate* an `OrderRepository`: a class with no domain analog, invented purely to keep `Order` cohesive.

This is the conceptual root of *repositories, services, mappers, validators* — all the supporting infrastructure that SRP and Clean Architecture eventually formalise.

=== 8. Indirection

[cols="1,4", options="header"]
|===
| | Statement

| *Problem*
| Where to assign a responsibility to avoid direct coupling between two or more things?

| *Solution*
| Assign the responsibility to an intermediate object to mediate between other components or services so they are not directly coupled.
|===

Insert a middle layer to decouple. `MediaPlayer` doesn't talk to `MP3Decoder` directly — it talks to a `Codec` interface, with `MP3Decoder` plugged in behind it. This is the principle behind *adapters, brokers, mediators, dependency-injection containers,* and the entire concept of *DIP*.

[quote, David Wheeler]
All problems in computer science can be solved by another level of indirection — except too many levels of indirection.

=== 9. Protected Variations

[cols="1,4", options="header"]
|===
| | Statement

| *Problem*
| How to design objects, subsystems, and systems so that the variations or instability in these elements does not have an undesirable impact on other elements?

| *Solution*
| Identify points of predicted variation or instability; assign responsibilities to create a stable interface around them.
|===

The most strategic pattern. Find the things you *expect* to change (database vendor, payment provider, currency, regulatory rules) and wrap them behind a stable abstraction so the rest of the system doesn't have to change when they do.

This is the meta-principle behind *OCP* and behind every plug-in architecture.

*Indirection is the _mechanism_. Protected Variations is the _intent_.*

== How GRASP maps to SOLID

[cols="1,2", options="header"]
|===
| GRASP | SOLID descendant

| High Cohesion | SRP
| Polymorphism | OCP, LSP
| Indirection + Protected Variations | DIP, OCP
| Low Coupling | implicit in all of SOLID
| Pure Fabrication | enables SRP, ISP
| Information Expert | no direct SOLID counterpart — orthogonal
| Creator | no direct SOLID counterpart — orthogonal
| Controller | thin-controller pattern, MVC, Clean Architecture use-cases
|===

== Spirit: GRASP vs. SOLID

* *GRASP is constructive* — "given this responsibility, which class should own it?"
* *SOLID is constraining* — "given this class, does it violate these rules?"

GRASP feels like *building*. SOLID feels like *auditing*. Used together, GRASP guides the original assignment and SOLID checks the result.

== Relation to other foundational concepts

* xref:ROOT:software-architecture/principles/SOLID.adoc[*SOLID*] — see mapping above.
* xref:ROOT:software-architecture/principles/coupling-and-cohesion.adoc[*Coupling & Cohesion*] — Low Coupling and High Cohesion are *literally* GRASP patterns; the foundational note generalises them beyond responsibility assignment.
* xref:ROOT:software-architecture/principles/KISS.adoc[*KISS*] — Pure Fabrication, Indirection, and Protected Variations all *risk* over-engineering. KISS reminds you to apply them only where the variation or coupling pressure is real.
* xref:ROOT:software-architecture/principles/YAGNI.adoc[*YAGNI*] — Protected Variations is the GRASP pattern most often misapplied. Build the protection only when the variation is real, not when it's imagined.
